Secure Examination System with Forward-Evolving Key Cryptography

Production-Grade Distributed Examination Platform

1. System Purpose

The Secure Examination System is a zero-trust, offline-capable examination platform designed for national-scale exams (JEE, NEET, CBSE, State Boards).

It enables encrypted examination content to be distributed days in advance to thousands of exam centers, while ensuring:

Before exam: content is mathematically undecryptable

During exam: instant local decryption within a fixed time window

After exam: permanent cryptographic irrecoverability

This enforcement is achieved without:

central servers during the exam

key revocation lists

administrative deletion

trusted clocks after unlock

Expiration is enforced purely through cryptographic key evolution.

2. Deployment Model

Each exam center receives a single static package:

exam.html        (~2.8 MB, signed)
exam_content.enc (~45 MB, encrypted)

Properties:

Zero installation

Runs in standard desktop browsers (Chrome / Edge / Firefox)

Fully offline during the exam

Identical package for all centers

3. Core Cryptographic Architecture (Key Design)
3.1 Key Roles

The system uses three cryptographic elements, each with a strict role:

Element	Purpose
SessionKey (K_session)	Encrypts the exam content
EvolutionKey₀ (K_evo₀)	Protects access to SessionKey
Evolution Function	Enforces irreversible time progression

There is:

One SessionKey

One WrappedSessionKey

No per-chunk keys

4. Authority Side (Encryption Phase)
Step 1: Generate Keys

The authority generates:

K_session → random 256-bit key

K_evo₀ → random 256-bit key

Step 2: Encrypt Exam Content

The exam PDF is chunked only for memory efficiency, not security.

For each chunk:

cipher_chunk_i = AES-256-GCM(K_session, chunk_i, unique_nonce_i)

All chunks share the same SessionKey.

Step 3: Wrap the Session Key (Critical Step)

The SessionKey is never stored or transmitted in plaintext.

Instead:

WrappedSessionKey = AES-KeyWrap(K_evo₀, K_session)
Distributed Package Contains:

Encrypted chunks

WrappedSessionKey

Evolution parameters (step count, rate)

Integrity signature (Ed25519)

5. Distribution Phase (Pre-Exam)

At exam centers:

Only encrypted data exists

WrappedSessionKey cannot be unwrapped

No decryptable material exists

The exam remains locked.

6. Exam Start (Unlock Phase)
Step 4: One-Time Unwrap

At the official exam start:

K_session = AES-KeyUnwrap(K_evo₀, WrappedSessionKey)

This happens exactly once.

Immediately after:

K_evo₀ derivation context is destroyed

Evolution enters forward-only mode

No re-unwrap is ever allowed

7. Exam Phase (Forward-Only Evolution)
Step 5: EvolutionKey Mutation

Every fixed interval (e.g., 30 ms):

K_evo_current = SHA3-256(K_evo_current)
counter++

Properties:

Forward-only

No rollback

No previous state recoverable

Independent of wall-clock

Step 6: Decryption During Exam

When content is needed:

plaintext_chunk_i = AES-GCM-Decrypt(K_session, cipher_chunk_i)

Because:

SessionKey is valid

EvolutionKey does not affect usage

UX remains smooth (scroll, revisit questions)

8. Exam End (Irreversibility Trigger)
Step 7: Counter Threshold Reached

When:

counter ≥ exam_threshold
Step 8: Accelerated Entropy Collapse

The system executes:

for i in 1..200,000:
    K_evo_current = SHA3-256(K_evo_current)

Effects:

EvolutionKey diverges irreversibly

WrappedSessionKey becomes permanently useless

No valid key material exists that can recover SessionKey

9. SessionKey Unreachability (Critical Guarantee)

After expiry:

SessionKey is never re-derived

No unwrap path exists

No regeneration mechanism exists

Any residual memory fragments are cryptographically unusable

Guarantee (Correctly Worded):

After the examination window, no cryptographic material exists that can be used to recover or apply the session key required to decrypt the examination content.

This is the correct and defensible claim.
